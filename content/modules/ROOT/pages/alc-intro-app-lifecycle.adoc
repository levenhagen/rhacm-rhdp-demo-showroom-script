= Introduction to Application Lifecycle
:navtitle: Introduction to Application Lifecycle

In the previous lab, you explored the Observability functionality in RHACM. This allowed you to monitor resources and prepare your system for potential disruptions.

Now, with governance and monitoring processes in place, the Application Lifecycle management functionality in RHACM provides the processes that are used to manage application resources on your managed clusters. This allows you to define a single or multi-cluster application using Kubernetes specifications, but with additional automation of the deployment and lifecycle management of resources to individual clusters. An application designed to run on a single cluster is straightforward and something you ought to be familiar with from working with OpenShift fundamentals. A multi-cluster application allows you to orchestrate the deployment of these same resources to multiple clusters, based on a set of rules you define for which clusters run the application components.

== Architecture & Concepts

In this context, GitOps is a set of practices for application configurations to expand upon existing processes and improve the application lifecycle. It uses Git repositories as a single source of truth to deliver applications declaratively. 

Today, there are a variety of ways to categorize GitOps topology models, however when considering models by intent there are two broad categories that we see: Centralized and Distributed. These represent different ArgoCD patterns, and each pattern have their benefits and it's up to the organization to decide what to prioritize. In summary:
In summary:

* *Centralized*: Single pane of glass but a single point of failure
* *Distributed*: No single point of failure, better scalability, however no single pane of glass.

image::alc-argo-cd-topologies.png[ArgoCD topologies]

There are variations on these models to address the identified weakness in each.

RHACM offers 3 different integration approaches for GitOps integration: 

* *Push model*: Traditional centralized approach with all ArgoCD components residing on the Hub.
* *Pull model*: Distributed approach, with some ArgoCD components moved spread across the managed clusters. 
* *Agent model*: Evolutionary approach that tries to accomplish the best of both worlds (Centralized and Distributed).
